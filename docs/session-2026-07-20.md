## Key decision — theme as data, not a second modal
Every project shared ONE modal (`#salonOverlay`) varying only by accent color. Instead of forking a second modal system, added a `theme` field to the project's DATA entry (`boba:{ …, theme:'boba' }`). `open()` toggles a `salon--boba` class when `theme==='boba'`, which (a) hides the baroque chrome (curtains/valance/tiebacks/filigree/motes/blinds/seal), (b) shows the `.boba-scene` layer, (c) restarts the pour via a `brew` class, and (d) fires `bobaOpenSound()` instead of the regal arpeggio. One pattern, cleanly extended; a 4th project theme is now trivial. Rejected: gooey-bloom, straw-punch, jelly-squish reveals (kept as options); a separate standalone modal (violated "improve the existing pattern, no new abstraction").
